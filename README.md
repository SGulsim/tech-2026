## 1. Общее описание

Цель проекта — разработка Telegram-бота для знакомств с системой анкет, рекомендаций и ранжирования пользователей.

Система позволяет:

- регистрировать пользователей через Telegram
- создавать и редактировать анкеты
- загружать фотографии в S3-совместимое хранилище
- просматривать анкеты других пользователей
- ставить лайки и пропуски
- формировать мэтчи
- инициировать общение после мэтча
- ранжировать анкеты по трёхуровневому алгоритму
- использовать реферальную систему
- кэшировать предварительно отранжированные списки анкет
- обрабатывать события асинхронно через очередь сообщений
- собирать метрики и вести логирование

## 3. Архитектура системы

Система построена по модульному принципу и состоит из следующих компонентов.

### 3.1 Компоненты

1. **Telegram Bot**
   - интерфейс взаимодействия с пользователем
   - обработка команд `/start`, `/profile`, просмотр анкет, лайки, пропуски
   - отправка запросов в backend

2. **Backend API (FastAPI)**
   - основная бизнес-логика
   - регистрация пользователей
   - CRUD анкет
   - обработка лайков, пропусков и мэтчей
   - работа с Redis, PostgreSQL, S3 и брокером сообщений

3. **PostgreSQL**
   - основное хранилище данных
   - хранение пользователей, анкет, лайков, мэтчей, рейтингов, рефералов

4. **Redis**
   - кэширование предварительно отранжированных списков анкет
   - ускорение выдачи рекомендаций пользователю

5. **Celery**
   - выполнение фоновых задач
   - регулярный и событийный пересчёт рейтингов
   - обработка асинхронных задач

6. **RabbitMQ**
   - передача событий лайков, пропусков, мэтчей и других действий
   - обеспечение асинхронного взаимодействия между сервисами

7. **S3 / MinIO**
   - хранение изображений пользователей
   - хранение ссылок на фото в базе данных

8. **Monitoring / Logging**
   - логирование действий пользователей и системных событий
   - сбор метрик производительности и активности

---

## 4. Взаимодействие компонентов

### Основной сценарий работы

1. Пользователь отправляет команду в Telegram-бот.
2. Бот передаёт запрос в Backend API.
3. Backend:
   - обрабатывает бизнес-логику
   - обращается к PostgreSQL
   - при необходимости использует Redis
   - загружает фото в MinIO
4. При событиях взаимодействия с анкетами:
   - backend публикует событие в RabbitMQ
   - Celery получает событие
   - выполняет пересчёт рейтинга или другие фоновые действия
5. При просмотре анкет backend использует кэшированные списки в Redis.

## 7. Алгоритм ранжирования

Система ранжирования строится на трёх уровнях.

### 7.1 Уровень 1: Первичный рейтинг

Формируется на основе данных анкеты пользователя:

- возраст
- пол
- интересы
- географическое положение

Также учитываются:

- полнота заполнения анкеты
- количество загруженных фотографий
- первичные предпочтения пользователя:
  - возрастной диапазон
  - пол
  - город

### 7.2 Уровень 2: Поведенческий рейтинг

Динамически корректируется на основе взаимодействий пользователей с анкетой:

- количество лайков анкеты
- соотношение лайков и пропусков
- частота взаимных лайков (мэтчей)
- частота инициирования диалогов после мэтча
- активность в определённое время суток

### 7.3 Уровень 3: Комбинированный рейтинг

Интегрирует первичный и поведенческий рейтинги по весовой модели.

Дополнительно учитывает:

- реферальную активность пользователя
- приглашение друзей в систему

### 7.4 Итоговая оценка

Финальный рейтинг вычисляется как агрегированная оценка:

- `level1_score` — первичный рейтинг
- `level2_score` — поведенческий рейтинг
- `level3_score` — комбинированный рейтинг
- `final_score` — итоговый балл для сортировки анкет

Пересчёт рейтингов выполняется:

- по событию взаимодействия
- по расписанию через Celery

## 8. Кэширование анкет (Redis)

Redis используется для ускорения выдачи рекомендаций.

### Принцип работы

- когда пользователь начинает сессию, первая анкета проходит полный путь обработки через backend
- одновременно backend подгружает, например, ещё 10 анкет
- эти анкеты сохраняются в Redis в предварительно отранжированном виде
- при дальнейшем просмотре бот получает анкеты из кэша
- когда пользователь доходит до последней анкеты из набора, цикл повторяется:
  - выполняется новая выборка
  - анкеты ранжируются
  - следующий пакет снова сохраняется в Redis

Такой подход снижает задержки при показе анкет и уменьшает нагрузку на базу данных.

## 17. Потоки данных

### Сценарий: просмотр анкет

1. Пользователь запрашивает следующую анкету.
2. Backend проверяет Redis:
   - если список есть, отдаёт следующую анкету из кэша
   - если списка нет, получает анкеты из БД, ранжирует их и сохраняет в Redis
3. При первой выдаче одновременно предзагружается дополнительный набор анкет.
4. Бот показывает анкету пользователю.

### Сценарий: лайк / пропуск

1. Пользователь ставит лайк или пропуск.
2. Backend:
   - записывает действие в БД
   - публикует событие в RabbitMQ
3. Celery:
   - получает событие
   - инициирует пересчёт рейтинга
   - при необходимости обновляет кэш

### Сценарий: мэтч

1. Если лайк взаимный, создаётся мэтч.
2. Backend сохраняет мэтч в БД.
3. Событие передаётся через RabbitMQ.
4. Notification Service отправляет уведомление пользователям.

### Сценарий: загрузка фото

1. Пользователь отправляет фото.
2. Backend загружает файл в MinIO.
3. Ссылка на изображение сохраняется в БД.
4. Обновляется `photo_count` профиля.

## 18. Локальный запуск

### Требования

- Docker и Docker Compose

### Шаги

1. Скопировать `.env.example` в `.env`:
   ```bash
   cp .env.example .env
   ```

2. Вставить токен бота в `.env`:
   ```
   TELEGRAM_BOT_TOKEN=your_token_here
   ```
